What was built

Drop-in Firebase adapter
src/api/base44Client.js was rewritten as a Firebase adapter that mirrors the exact Base44 SDK interface — .list(), .create(), .update(), .delete(), .filter(), .subscribe(), .auth.*, .functions.invoke(). Zero changes required to any of the 52 existing page files.
Firebase initializer
src/api/firebase.js initializes the Firebase app and exports db (Firestore), auth (Firebase Auth), and functions (Cloud Functions). Reads config from VITE_FIREBASE_* environment variables.
Auth context
src/lib/AuthContext.jsx rewritten using Firebase's onAuthStateChanged. User profiles are stored in a users/{uid} Firestore document and created automatically on first sign-in.
Login page
src/pages/Login.jsx — new login page with Google Sign-In and email/password support. Styled to match the app's dark theme. Handles post-login redirect automatically.
Cloud Functions
functions/index.js — all 5 backend functions migrated from Deno/Base44 to Firebase Cloud Functions (Node.js): spotifyTrackLookup, youtubeChannelLookup, fetchCompetitorData, fetchTourOpportunities, analyzeSong, sendWelcomeEmail.
Firestore security rules
firestore.rules — per-user data scoped by user_id field. Shared collections (CommunityMessage, ReportCollaborator, ReportComment) are readable/writable by any authenticated user.

To go live — step by step

1
Create a Firebase project
Go to console.firebase.google.com, create a new project, and enable Firestore and Authentication. In Authentication → Sign-in method, enable Google and Email/Password.
2
Add your Firebase config to .env.local
In Firebase Console → Project Settings → Your apps → Web app, copy the config object. Create .env.local in the project root: 
VITE_FIREBASE_API_KEY=AIzaSy...
VITE_FIREBASE_AUTH_DOMAIN=your-project.firebaseapp.com
VITE_FIREBASE_PROJECT_ID=your-project-id
VITE_FIREBASE_STORAGE_BUCKET=your-project.appspot.com
VITE_FIREBASE_MESSAGING_SENDER_ID=123456789
VITE_FIREBASE_APP_ID=1:123456789:web:abc123
3
Update .firebaserc with your project ID
Open .firebaserc and replace YOUR_FIREBASE_PROJECT_ID with your real project ID (same value as VITE_FIREBASE_PROJECT_ID).
4
Install Firebase CLI and Functions dependencies
# Install Firebase CLI globally (if not already installed)
npm install -g firebase-tools

# Log in
firebase login

# Install Cloud Functions dependencies
cd functions && npm install && cd ..
5
Set Cloud Functions environment secrets
Each function uses API keys stored as Firebase secrets. Set them with: 
# Spotify — https://developer.spotify.com/dashboard
firebase functions:secrets:set SPOTIFY_CLIENT_ID
firebase functions:secrets:set SPOTIFY_CLIENT_SECRET

# YouTube Data API v3 — https://console.cloud.google.com
firebase functions:secrets:set YOUTUBE_API_KEY

# Anthropic (AI features) — https://console.anthropic.com
firebase functions:secrets:set ANTHROPIC_API_KEY

# Email — optional, for welcome emails (Gmail app password or SMTP service)
firebase functions:secrets:set SMTP_USER
firebase functions:secrets:set SMTP_PASS
6
Deploy to Firebase
# Deploy everything (Firestore rules + Cloud Functions + Hosting)
npm run build
firebase deploy

# Or deploy individually
firebase deploy --only firestore:rules
firebase deploy --only functions
firebase deploy --only hosting
7
Migrate existing data from Base44
Export your data from the Base44 dashboard (Settings → Export). Each entity collection maps 1-to-1 with a Firestore collection of the same name. You'll need to add a user_id field (your Firebase UID) to each document before importing. A simple migration script can do this.
8
Test locally with the Firebase emulator
For local development without deploying: 
# Start all emulators (Auth, Firestore, Functions, Hosting)
firebase emulators:start

# In another terminal, start the Vite dev server
npm run dev
Note: The emulator UI runs at http://localhost:4000 — you can browse your Firestore data, manage users, and view function logs there.

Data model

26 entities → 26 Firestore collections
Every entity in base44/entities/ maps directly to a Firestore collection with the same name. Documents include a user_id field for ownership, plus created_date and updated_date ISO strings on every write.
User profiles
Each user's profile (artist name, bio, role, onboarding state, etc.) is stored in users/{uid}. Created automatically on first sign-in. Updated via base44.auth.updateMe() — same API as before.
Shared collections
CommunityMessage, ReportCollaborator, and ReportComment are not scoped by user — they're accessible to all authenticated users, matching the original collaborative behavior.
Filtering & sorting
All data is fetched per-user with a single Firestore where('user_id', '==', uid) query, then sorted and filtered client-side. This avoids composite index requirements and works out of the box.
